# Architecture Decision Records (ADR)

이 디렉토리는 프로젝트의 아키텍처 결정 사항을 기록한 **ADR(Architecture Decision Record)** 을 보관하는 곳입니다.

## ADR이란?

ADR은 중요한 아키텍처 관련 결정을 문서화하는 간결한 형식의 기록입니다.  
각 결정의 배경, 선택한 대안, 결과 등을 명확히 남겨 팀원과 이해관계자가 결정의 맥락을 이해하고 향후 참고할 수 있도록 합니다.

## 왜 ADR을 작성하는가?

- **결정의 투명성**: 왜 특정 기술이나 패턴을 선택했는지 그 이유를 기록합니다.
- **지식 공유**: 팀원 간 의사 결정 과정을 공유하고, 새로 합류한 구성원도 과거 결정을 쉽게 파악할 수 있습니다.
- **회고와 개선**: 나중에 결정을 재검토하거나 변경해야 할 때 근거를 명확히 알 수 있습니다.
- **일관성 유지**: 유사한 상황에서 동일한 기준으로 판단할 수 있는 기준점이 됩니다.

## ADR 작성 가이드라인

- 각 ADR은 `docs/adr/{번호}-{제목}.md` 형식으로 저장합니다.  
  예: `001-cni-selection.md`
- 번호는 001부터 순차적으로 부여하며, 한 번 발행된 번호는 변경하지 않습니다.
- 모든 ADR은 아래 템플릿을 따릅니다.

## ADR 템플릿

```markdown
# ADR-번호: [제목]

## 상태
[Proposed | Accepted | Deprecated | Superseded by ADR-xxx]

## 컨텍스트
이 결정이 필요한 배경과 관련된 정보를 서술합니다.
현재 시스템 구조나 기술 상황, 외부 제약 조건 등을 기술합니다.

## 결정
어떤 결정을 내렸는지를 명확하게 서술합니다.
선택한 아키텍처나 도구, 패턴, 구조 등과 그 이유를 기술합니다.

## 근거
왜 이 결정을 내렸는지 설명합니다.
다른 대안들과 비교한 장단점, 주요 고려 요소 등을 포함합니다.

## 결과
이 결정이 시스템에 미치는 영향이나, 향후 따라야 할 구현 및 운영 지침을 기술합니다.
추가로 생기는 작업이나 영향 범위 등도 포함됩니다.

## 대안
검토했던 대안들과 선택하지 않은 이유를 요약합니다.

## 작성일자
YYYY-MM-DD
```
